### PRJ_01 - 晶圆制造缺陷分析系统

| 属性 | 详情 |
|------|------|
| **时间** | 2024.09 - 2025.07 |
| **角色** | AI产品经理 / 全栈开发 |
| **公司/团队** | 浙江大学 · 三脚架队（3人团队）|
| **地点** | 杭州 |
| **状态** | 已上线（校企合作） |
| **标签** | Python, ResNet50, CLIP, YOLOv8, SAM, React, Flask |

#### 项目概述
针对产线良率分析依赖人工、缺乏根因追溯的痛点，主导从 0 到 1 设计"检测-诊断-溯源"闭环系统。与校企合作公司合作，使用真实产线数据训练，将模型能力封装供员工调用分析。

#### 核心贡献

**产品设计**
- 主导从 0 到 1 设计"检测-诊断-溯源"闭环系统，撰写完整 PRD 文档
- 定义四大核心模块：缺陷检测、缺陷分割、缺陷分类、根因分析
- 完成交互原型开发与全栈集成

**全栈开发**
- 前端：React + Tailwind CSS，设计并实现高保真 UI 界面
- 后端：Flask 搭建 API 服务，连接算法模型与前端交互

#### 技术架构

**四大核心模块**
| 模块 | 技术 | 功能 |
|------|------|------|
| 缺陷检测 | YOLOv8 | 快速定位晶圆图中的缺陷位置 |
| 缺陷分割 | SAM | 精确分割缺陷边界，计算面积占比 |
| 缺陷分类 | ResNet50 | 12类晶圆缺陷分类，Top-1准确率96.2% |
| 根因分析 | CLIP+OCR | 视觉问答+知识库匹配，输出缺陷成因 |

**技术难点与解决方案**
- **晶圆图与自然图像分布差异大**：晶圆图具有复杂的规律性背景图案，缺陷尺度差异大，传统视觉模型效果差
- **工业少样本问题**：仅2000+张标注数据，通过迁移学习+数据增强解决
- **实时性要求**：推理速度优化至0.03秒，满足产线实时检测需求

#### 技术亮点
- **多模态检索**：设计 CLIP 视觉问答策略，实现 Top-1 准确率 96.2%及 0.03 秒实时推理
- **闭环系统**：从缺陷发现到根因追踪的全流程自动化
- **校企合作落地**：使用真实产线数据训练，模型能力已封装供员工调用

- 如果后续用于面试，建议统一一版口径：到底是“检测/分割/分类/根因分析”，还是“分类/分割/语义诊断/以图搜图”

#### 2026-03-11 面试问答沉淀
- 目标用户已明确为半导体产业的 YE（良率）工程师，其核心职责是分析缺陷成因、监控产线状态、调整生产 recipe 或反馈机台问题，从而把控整体生产良率
- 项目使用场景已明确为：系统接入扫描电镜机台拍摄的缺陷图像，对单个缺陷完成分类、分割、语义描述与历史根因库检索，并输出一份多维度辅助分析报告
- 当前统一口径应调整为四模块：`分类 / 分割 / 语义 / 检索`
- 四模块当前输入统一为缺陷 SEM 图；其中机台实际会对同一缺陷拍摄三张图，分析前先将三张图作为 RGB 三通道合成为一张图
- 当前各模块输出已明确：
- 分类输出：缺陷类别文本
- 分割输出：二值化分割结果
- 语义输出：五句语义描述
- 检索输出：Top 10 相似图像，且每张图都可进一步查询数据库中记录的历史根因
- 当前系统的总输出不是“自动根因结论”，而是“辅助信息汇总报告”；原因是语义材料受限、根因数据量不足，尚不支持稳定的一站式根因推断
- 理想业务流为：系统自动抓取扫描电镜机台拍摄的全部 SEM 图，对每个缺陷自动完成四模块分析并生成辅助报告，供 YE 工程师人工分析时联动查阅
- 这意味着该项目更准确的定位是“YE 缺陷分析辅助系统”，而不是“完全替代人工的自动根因诊断系统”
- 仍待进一步问清的问题：
- 机台原始流程里，单个缺陷从发现到 YE 查看报告，中间经过哪些系统和人工节点
- 四模块分别采用了哪些模型或规则，以及语义描述具体是如何生成的
- 为什么要把三张 SEM 图合成 RGB 三通道，而不是多视角独立分析后再融合
- 报告最终长什么样，YE 会重点看哪些字段，哪些字段真正影响他们后续动作

#### 复盘总结
- ✅ **商业价值落地**：与校企合作，使用真实产线数据，将人工标注成本降低 90%
- ✅ **技术与业务平衡**：充分考虑精度与产线效率的平衡，确保系统实用性
- ✅ **产品思维实践**：从需求调研到产品交付的完整流程，积累了AI产品化经验

### 1. 页面结构
- 当前项目是一个基于 React + Vite 的单页作品集，顶层分为 `Home`、`Projects`、`About`
- 页面状态由 `App.tsx` 控制，点击首页精选卡片或项目列表卡片会进入项目详情页
- `Projects` 页支持按 `ALL / PROJECT / INTERNSHIP` 筛选，详情页支持同类项目的上一个/下一个切换

### 2. 数据组织方式
- 项目与实习经历统一维护在 `src/data.ts` 的 `experiences` 中
- 每条经历都被结构化成 `overview / contributions / highlights / metrics / retrospective / images / demoUrl`
- 这意味着 `PROJECTS.md` 最适合持续沉淀的内容，不只是项目描述，还包括亮点、指标、复盘和展示素材

### 3. 当前展示策略
- 首页精选项为 `PRJ_01`、`EXP_01`、`PRJ_02`，说明当前作品集默认叙事主线是“硬核项目 -> To B 实习 -> 可演示 AI 产品”
- `PRJ_01` 是当前最完整的展示条目，有图片、有指标、有落地结果
- `PRJ_02` 是当前最适合演示交互的条目，因为有内嵌 demo
- `EXP_01` 和 `PRJ_03` 更偏“经历与能力证明”，但还缺少视觉素材支撑

### 4. 当前可见问题
- `src/data.ts` 以及部分组件文案存在中文编码异常，终端中已明显出现乱码；这不影响本次 `PROJECTS.md` 梳理，但后续建议单独修复
- 当前很多项目没有配图，导致详情页展示力不均衡
- `PROJECTS.md` 中已有较完整项目信息，下一阶段更值得补的是“可追问细节”和“统一口径”
